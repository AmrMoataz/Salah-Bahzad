# Native App · A2 — WIRING stream (the on-device manual capture-protection matrix: screenshot + screen-record render black, per OS)

> Status: **Planned — not yet built** · Created 2026-06-26 · Proves phase **A2** end-to-end **on real devices** — the running app, a real screenshot, and a real screen-recording on each OS. This is the phase whose **"live check" is a HUMAN, not HTTP**: `NFR-APP-MAINT-003` mandates "a documented manual test matrix per OS (screenshot + screen-record on each), executed before release", and this stream **is** that matrix. The A2 headline: while a lesson is on screen, **a screenshot AND a screen-recording both render black** on every platform that can guarantee it (Android, Windows, macOS), **iOS defeats recording and flags+logs the screenshot** (the accepted gap, `NFR-APP-CAP-005`); protection **provably precedes the first frame** (`NFR-APP-CAP-006`) and **clears on leaving the player** (`FR-APP-CAP-003`); and an **unsupported config warns-and-refuses** rather than playing unprotected (`NFR-APP-COMPAT-002`). **No backend and no contract change in the whole slice** (master plan §A2, line 205: `Backend: none`) — there is **no contract drift to check** because A2 touches neither the backend nor `docs/contracts/native-app-playback.md`.
>
> Runs **after** the A2 app stream (the `core/secure_surface` shims) merges. ⚠️ **BUILD CONSTRAINT (read this first):** **Android + Windows are verifiable from this Windows dev box** — Windows natively (`flutter run -d windows`), Android on an emulator or USB device (`flutter run -d <android>`). **macOS + iOS REQUIRE a Mac**: the Swift shims (`MainFlutterWindow.swift`, `AppDelegate.swift`) do not compile on Windows, and **iOS additionally needs a physical device** (the Simulator doesn't faithfully reproduce `UIScreen.isCaptured`) plus a **development signing identity** — a **free Apple ID** provisions the debug build onto a device for this capture test; the **paid Apple Developer Program account (`NFR-APP-DIST-001`) and signed packaging are an A4 concern, not A2** — both the **build** and the **manual capture** for macOS/iOS are a Mac-side step, recorded in the matrix as **pending — Mac required**. To reach a *real protected frame* to attempt to capture, reuse the **A1** live path (run the API via AppHost, mint a Student token, seed a **Ready** AES-128 HLS video + active enrollment) — but the A2 proof itself is **purely on-device observation**, not an HTTP call.
> Satisfies (proven on-device): `FR-APP-CAP-001` (screenshots + recordings render black while video is on screen), `FR-APP-CAP-002` (active capture/mirroring → pause/blank, resume when it stops), `FR-APP-CAP-003` (enabled before the first frame, disabled only on leaving the player), `NFR-APP-CAP-001` (Android `FLAG_SECURE` → black + excluded from recent-apps thumbnail), `NFR-APP-CAP-002` (Windows `WDA_EXCLUDEFROMCAPTURE` → excluded from OBS/Snipping Tool), `NFR-APP-CAP-003` (macOS `sharingType = .none`), `NFR-APP-CAP-004` (iOS `UIScreen.isCaptured` → blank/pause), `NFR-APP-CAP-005` (iOS still-screenshot accepted gap → `userDidTakeScreenshot` log/flag + watermark), `NFR-APP-CAP-006` (protection precedes the first frame — no unprotected window of exposure), `NFR-APP-COMPAT-001` (min OS floors so the APIs exist), `NFR-APP-COMPAT-002` (warn-and-refuse where the black-out can't be guaranteed — fail safe, not silent), `NFR-APP-MAINT-003` (the documented per-OS manual matrix itself), `FR-APP-VID-003` (the serial+name watermark the iOS screenshot gap falls back on).

## Design reference
Verifies behaviour, not pixels. The on-screen surface A2 makes *truthful* is the Player's **"Screen capture blocked"** banner — anchor on the prototype `PLAYER` banner + its **"Preview black-out"** affordance (`.claude/Salah Bahzad App/Secure Video App (standalone).html`; master plan §A2 line 206). **Not** the Teacher Portal prototype, **not** `docs/tokens.*`. A1 shipped that banner as a **static string** (`app/lib/features/player/widgets/capture_blocked_banner.dart`, a parameterless `StatelessWidget` mounted at `app/lib/features/player/player_view.dart:92-97`); A2 wires the real OS black-out behind it and (app stream) drives the banner from the real `SecureSurfaceStatus` — `protected` (green) / `unsupported` (amber "best-effort, can't block on this device") / `off`. The entire on-device walkthrough (#18) is the **user's step**.

## AppHost change to land first
**None.** A2 adds no backend, no migration, no Aspire service. To reach a real playing lesson worth attempting to capture, run the API exactly as in A1 — `dotnet run` the AppHost (F5), then run the app pointed at the stable named `app` endpoint:
- **Windows (this box):** `flutter run -d windows --dart-define=API_BASE_URL=http://localhost:5080`
- **Android (this box, emulator/device):** `flutter run -d <android-device> --dart-define=API_BASE_URL=http://10.0.2.2:5080` (emulator loopback alias) or the LAN IP for a physical device.
- **macOS / iOS (Mac required):** on the Mac, `flutter run -d macos` / `flutter run -d <ios-device> --dart-define=API_BASE_URL=http://<mac-LAN-ip>:5080`.

The app is native (no browser) → **no CORS**. TLS is never disabled — dev is HTTP on the AppHost `app` endpoint on purpose (Dart can't trust the local dev cert); prod is HTTPS with a real CA cert. Register `salah-bahazad://` locally to fire a Play link (HKCU on Windows → debug exe `"%1"`; trigger with `start "" "salah-bahazad://stream?videoId=…&sessionId=…&handoff=…"`). Real HLS needs `ffmpeg` resolvable to the API (AppHost `ResolveFfmpeg` → `Transcode__FfmpegPath`) so the fixture video is genuinely **Ready** — reuse the **A1** real-ffmpeg fixture.

## Live checks (the manual capture matrix — every step is an on-device observation, NOT an HTTP call)
> For each platform: **(a)** prove protection is **on before the first frame**; **(b)** take a **still screenshot**; **(c)** take a **screen-recording** (incl. mirroring on mobile); **(d)** assert **both render black** on the platforms that guarantee it (iOS: recording black, screenshot **flagged+logged**, not blocked); **(e)** leave the player and prove protection **clears**. Fill the matrix table below as you go.

**Android — `FLAG_SECURE` (#1–#4, `NFR-APP-CAP-001`, `FR-APP-CAP-001/002/003`, builds here):**
1. **Precedes the first frame** (`NFR-APP-CAP-006`): mount the Player; confirm the Kotlin shim's `enable` ran via `window.addFlags(WindowManager.LayoutParams.FLAG_SECURE)` (registered in `app/android/app/src/main/kotlin/com/salahbahzad/secure_player/MainActivity.kt` via `configureFlutterEngine`) **before** the engine opens (`app/lib/features/player/player_controller.dart:89`, `_engine.open`). Tell-tale: the **recent-apps thumbnail is already black** the instant the route paints — there is no unprotected window.
2. **Screenshot → black:** with the lesson playing, take a system screenshot → the saved image is **black** (only chrome/system bars, no video).
3. **Screen-record → black:** record the screen for a few seconds of playback → the recording shows the **blacked-out** surface throughout; the app is **excluded from the recent-apps thumbnail** too (`NFR-APP-CAP-001`).
4. **Clears on leaving** (`FR-APP-CAP-003`): back out of the Player → the shim's `disable` runs `clearFlags(FLAG_SECURE)` in `dispose`; a screenshot of **/idle** (home) is now **normal**, proving the flag did not leak past the player.

**Windows — `SetWindowDisplayAffinity(WDA_EXCLUDEFROMCAPTURE)` (#5–#8, `NFR-APP-CAP-002`, builds here):**
5. **Precedes the first frame:** mount the Player; confirm the C++ shim called `SetWindowDisplayAffinity(GetHandle(), WDA_EXCLUDEFROMCAPTURE)` on the Flutter view `HWND` (registered after `app/windows/runner/flutter_window.cpp:27`) **before** `player_controller.dart:89` opens the engine.
6. **Screenshot → excluded:** `PrintScreen` / Snipping Tool / Win+Shift+S over the playing window → the captured image shows the window as **black / excluded** (the rest of the desktop captures normally).
7. **Screen-record → excluded:** record with **OBS** (Display Capture) and the Xbox Game Bar → the player window is **excluded from the recording** (black or absent) for the whole clip (`NFR-APP-CAP-002`: OBS, Snipping Tool, etc.).
8. **Clears on leaving:** close the Player → the shim resets `SetWindowDisplayAffinity(hwnd, WDA_NONE)`; the idle window now captures **normally** in both a screenshot and OBS.

**macOS — `NSWindow.sharingType = .none` (#9–#12, `NFR-APP-CAP-003`, ⚠️ MAC REQUIRED):**
9. **Precedes the first frame:** on the Mac, mount the Player; confirm the Swift shim set `self.sharingType = .none` on the `NSWindow` (registered after `app/macos/Runner/MainFlutterWindow.swift:11`) before the engine opens.
10. **Screenshot → black:** `Cmd-Shift-4` window screenshot of the player → the saved image is **black** (the window is excluded from window screenshots, `NFR-APP-CAP-003`).
11. **Screen-record → excluded:** `Cmd-Shift-5` screen recording + a screen-share session → the player window is **excluded from screen sharing/recording** throughout.
12. **Clears on leaving:** close the Player → the shim restores `sharingType = .readOnly` (the non-protected default — **not** `.none`); a subsequent window screenshot/recording of idle is **normal**.

**iOS — `UIScreen.isCaptured` + `userDidTakeScreenshot` (#13–#16, `NFR-APP-CAP-004/005`, ⚠️ MAC + PHYSICAL DEVICE + DEV SIGNING REQUIRED):**
13. **Precedes the first frame:** on the iPhone, mount the Player; confirm the Swift shim attached the KVO/`UIScreen.capturedDidChangeNotification` observer (streamed over the `EventChannel`, registered in `app/ios/Runner/AppDelegate.swift:13-15`) before the engine opens.
14. **Recording / mirroring → blank-pause-resume** (`NFR-APP-CAP-004`, `FR-APP-CAP-002`): start Control-Center **Screen Recording** (and separately, **AirPlay mirroring** to another screen) while playing → playback **blanks/pauses**, the recording shows the **blanked** state, and playback **resumes only when capture stops**.
15. **Screenshot → ACCEPTED GAP** (`NFR-APP-CAP-005`): take a still screenshot → it is **NOT blocked** (iOS has no single-screenshot block without FairPlay, which is **deferred, not v1**). Assert instead the two-part honest gap: **(a)** the app **detected it via `userDidTakeScreenshot` and logged/flagged the event** (verify the log/flag fired — no secrets logged), and **(b)** the captured frame carries the **visible watermark** — `"{serial} · {fullName}"`, `FR-APP-VID-003` — so the leak is **traceable**. Recording stays blocked by #14.
16. **Clears on leaving:** close the Player → the observer is removed; later screen recording outside the player is **not** blanked, proving the protection is player-scoped.

**COMPAT — warn-and-refuse, never silent (#17, `NFR-APP-COMPAT-001/002`):**
17. On a config where the black-out **can't be guaranteed** — the canonical case is **Windows 10 < 2004 (build < 19041)** where `WDA_EXCLUDEFROMCAPTURE` is unavailable — `isSupported`/`enable` returns `SecureSurfaceStatus.unsupported`; the app **warns** (the banner flips to the amber "best-effort / can't block on this device" state) **and REFUSES protected playback** rather than playing unprotected silently (`NFR-APP-COMPAT-002`: "Fail safe, not silent"). Confirm the documented **min OS floors** (`NFR-APP-COMPAT-001`) are enforced/declared: **Windows 10 2004+, macOS 11+, iOS 13+, Android 8+**. (If no genuine old build is reachable, force the `unsupported` reply via a debug override and **note it was forced** in the log.)
